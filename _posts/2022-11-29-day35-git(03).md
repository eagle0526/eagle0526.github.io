### git pull 線上抓下新檔案

如果今天線上進度比本機還新，要怎麼把檔案抓下來呢

```md
> 本機                github
> 
>                     口   git在這   
> 
> 口 本機進度在這       口
> 
> 口                  口
> 
> 口                  口
```

接下來用fetch指令，他會在本機抓線上最新的檔案回來
並且在本機形成一個遠端分支，origin/main為啥叫這個名字呢。因為遠端分支是origin，本機分支是main

```md
> 本機                github
> 
> 口 本機要更新到這     口   git在這   
>    origin/main
> 口                  口
> 
> 口                  口
> 
> 口                  口
```


把遠端分支最新的檔案抓下來到本機
```md
> git fetch origin main
```

把本機的main跟origin分支融合，就可以
```md
> git merge origin/main
```

```md
> git pull = git fetch + git merge
```

> --    
> 如果今天git pull有衝突，是git merge的衝突喔，因為純粹fetch拉下來，不會有錯誤  
> 錯誤是發生在東西合併
> --    
{: .block-tip}

PUll Request(PR) 
------

假設原作同意收下這個pr，就代表


先從指定的github，拉一份檔案下來(直接用fork就好)

然後更改裡面檔案的資料(直接在github上改就好)

接著按下PR的按鈕，就可以


> --    
> 之後團隊合作，都是用PR的方式來協作    
> --    
{: .block-tip}



### 主要擁有者的看PR環節

如果同意接收變動，那就選取merge

點files changed
就可以看到這個PR改了哪些檔案
並且可以直接對這個PR評論


評論完後，該則改則檔案變動就會變成pending狀態

有一個approve按鈕，只是單純認同你的更動，不代表是直接merge
有一個request按鈕，是要求你改你的更動




三種收pr的方式 - 假設原本的commit數量有2個
```md
> create a merge commit -> 會有一個新的節點(對方發的commit有兩個，收完之後變五個)
> rebase and remote     -> 不會有節點(對方發的commit有兩個，收完之後變四個)
> squash and merge      -> 全部搜集成一個commit(對方發的commit有兩個，收完之後變三個)
```

> --    
> 如果PR衝突怎麼辦，主辦人退回給寫PR的人，叫他改過再來  
> --    
{: .block-tip}

> --    
> 什麼時候會用到squash？就是發PR的人寫太多無用訊息  
> --    
{: .block-warning}



> --    
> **Approve的效用**  
> 可以設定，每一個PR有多少人Approve，才可以merge進去專案裡面   
> 所以在做專案的時候，如果團隊有五個人，一定要設定approve是四個  
> 這樣代表大家都要同意這件事，才能讓專案推上去      
> --    
{: .block-tip}
